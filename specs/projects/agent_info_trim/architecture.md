## Helper changes

### New shared helper: `truncate_to_words_with_agent_sentinel`

Location: `libs/core/kiln_ai/utils/formatting.py`, alongside `truncate_to_words`.

```python
AGENT_TRUNCATION_SENTINEL = "[...truncated, load task for full instructions]"

def truncate_to_words_with_agent_sentinel(
    text: str | None, max_words: int
) -> str | None:
    """Truncate to max_words and append AGENT_TRUNCATION_SENTINEL when truncation
    occurred. Preserves None for None. Applied to task instructions surfaced to
    the LLM agent — see specs/projects/agent_info_trim."""
    truncated, was_truncated = truncate_to_words(text, max_words)
    if not was_truncated:
        return truncated
    # truncate_to_words currently appends " …" when truncated; strip it, then
    # strip trailing whitespace, then append the sentinel.
    stripped = truncated.removesuffix(" …").rstrip()
    return f"{stripped} {AGENT_TRUNCATION_SENTINEL}"
```

Implementation note: `truncate_to_words` today appends ` …`. We strip that suffix to avoid doubled ellipsis-plus-sentinel. If the existing helper behaviour is ever changed to not append an ellipsis, the `removesuffix` call becomes a no-op — not a correctness risk.

The existing `truncate_to_words` is **not modified** — other callers in the codebase (non-agent paths) may rely on its current behaviour.

### `_prompts_block` — narrowed source pool + top-5 selection

Rewritten to:

1. Compute `generators_from_task_instruction_count = len(prompt_generators)` — **do not** emit generators into `items`.
2. Gather the "real" prompt pool — a list of `(id, name, type_label, created_at)` tuples:
   - Persisted prompts from `task.prompts(readonly=True)`: `(f"id::{p.id}", p.name, prompt_type_label(..., p.generator_id), p.created_at)`.
   - Fine-tune virtuals from `task.finetunes(readonly=True)` with `fine_tune_model_id is not None`: `(f"fine_tune_prompt::{project.id}::{task.id}::{ft.id}", ft.name, prompt_type_label(..., None), ft.created_at)`.
   - Task-run-config frozen virtuals from `task_run_configs` where `rc.prompt is not None`: `(f"task_run_config::{project.id}::{task.id}::{rc.id}", rc.name, prompt_type_label(..., rc.prompt.generator_id), rc.created_at)`.
3. `total = len(pool)`.
4. `items = _select_recent(pool, cap=5)` — sort by `(created_at, id)` descending, take first 5.
5. `showing = f"{len(items)} of {total}"`.

Signature stays `(task, project, task_run_configs) -> AgentOverviewPrompts`.

### `_run_configs_block` — starred-first selection

Rewritten to:

1. `total = len(task_run_configs)`.
2. Split into `starred` and `unstarred` lists; sort each by `(created_at, id)` desc.
3. If `len(starred) >= 5`: `selected = starred` (all of them, no cap).
4. Else: `selected = starred + unstarred[:5 - len(starred)]`.
5. Map `selected` → `AgentOverviewRunConfig` items (same mapping logic as today, minus `created_at`).
6. `showing = f"{len(items)} of {total}"`.

Signature unchanged: `(task, task_run_configs) -> AgentOverviewRunConfigs`.

### Small reusable selection helper

Factor the top-N-by-recency logic into one private helper to avoid duplication between `_prompts_block` and `_run_configs_block`:

```python
def _top_n_by_recency(
    items: Iterable[T],
    *,
    key: Callable[[T], tuple[datetime, str]],
    cap: int,
) -> list[T]:
    return sorted(items, key=key, reverse=True)[:cap]
```

Used twice; not exported. Keeps the ordering rule ("`created_at` desc, tie-break on id desc") defined in one place so both sections behave identically.

### `_fine_tunes_block` / `_prompt_optimization_jobs_block`

Replaced with count-only producers:

```python
def _fine_tunes_block(task: Task) -> AgentOverviewFineTunes:
    return AgentOverviewFineTunes(
        total_count=sum(1 for _ in task.finetunes(readonly=True))
    )

def _prompt_optimization_jobs_block(task: Task) -> AgentOverviewPromptOptimizationJobs:
    return AgentOverviewPromptOptimizationJobs(
        total_count=sum(1 for _ in task.prompt_optimization_jobs(readonly=True))
    )
```

The `prompt_optimization_jobs_block` no longer needs `run_configs_by_id` (used previously to resolve model name/provider) — simplified signature.

Note: `_prompts_block` still reads `task.finetunes(readonly=True)` for fine-tune virtual prompts. That's a separate iteration from `_fine_tunes_block`. Acceptable — lists are small; keeping the two concerns separate is clearer than threading a shared list through both helpers.

## Key design decisions

### 1. Sentinel helper at core `utils/formatting` level

The sentinel is an agent-LLM-contextual detail — but it's low-level string formatting reused by two endpoints. Putting it next to `truncate_to_words` keeps a single coherent home for truncation utilities and gives `task_api.py` (which lives in `libs/server`, without a dependency on `app/desktop`) access to it. The alternative (keep it in `agent_api.py` and import across layers) would create a bad dependency direction.

### 2. Keep `truncate_to_words` unchanged; layer the sentinel on top

The existing helper has other callers (search the codebase — non-agent-facing truncation uses `… ` suffix today). Modifying its signature or behaviour breaks them. A thin wrapper is simpler and leaves the existing contract intact.

### 3. New `Prompts` / `RunConfigs` containers — breaking shape change

The prior release shipped together (agent_api_info), so we're not worrying about backwards compatibility here — per the functional-spec "Goals" section, coordinated updates are acceptable. All consumers are regenerated TS types plus the agent's generic `call_kiln_api` tool, which handles arbitrary JSON.

### 4. Selection helper factored once

Top-N-by-recency logic is simple (`sorted(...)[:cap]`), but factoring it out forces both prompts and run_configs to use the exact same tiebreak rule (`(created_at, id) desc`). Without the helper, drift between the two sections is easy to introduce in a later edit.

### 5. No partial-response caching

As in the parent spec: every request recomputes. Datamodel-level caches are sufficient. No new caches introduced.

### 6. `_prompts_block` receives the narrow pool; no shared state with `_fine_tunes_block`

Two separate iterations over `task.finetunes()` is fine (small, memory-cached). Cleaner signatures than threading a shared list through both helpers.

### 7. `generators_from_task_instruction_count` computed at request time

Uses `len(prompt_generators)` from `libs/server/kiln_server/prompt_api.py`. If a future change adds a generator, the count updates automatically with no code change here.

### 8. Endpoint rename is a hard swap, not dual-routing

The parent spec is only a few weeks old; no external consumers beyond the chat agent and generated TS types. Double-routing `/api/all_tasks` alongside `/api/task_summaries` would add a dead path and muddy the docstring rationale. Delete the old path.

## Error handling

Unchanged from parent spec:

| Case | Response |
|---|---|
| Unknown project_id | 404 from `task_from_id` |
| Unknown task_id | 404 from `task_from_id` |
| Empty task (no runs, no prompts, etc.) | 200 OK; counts are zero; `items` lists empty; `generators_from_task_instruction_count` still non-zero |
| File-read errors during `task_summaries` project load | Log + skip project (existing behaviour, unchanged) |
| Legacy `/api/all_tasks` path | 404 (FastAPI default for unknown routes) |

### Formatting helper (`libs/core/kiln_ai/utils/test_formatting.py`)

  - Under limit → no sentinel, no change.

- `run_configs` selection:
  - 8 starred, 5 unstarred → all 8 starred returned, unstarred padding skipped, `total == 13`.
  - 2 starred, 10 unstarred → 2 starred + 3 most-recent unstarred = 5 items, `total == 12`.
  - 0 starred, 3 unstarred → 3 items, `total == 3`.
  - 4 starred, 0 unstarred → 4 items, `total == 4`, `showing == "4 of 4"`.
- `run_configs` ordering: within the starred prefix, more-recent first; padded unstarred follow, more-recent first.
- `task.instruction` truncation at exactly 70 words: sentinel absent; one word over: sentinel present and appears exactly once.
