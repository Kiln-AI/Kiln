# Sub-agents for Kiln Assistant

## Context

The Kiln Assistant (desktop app ↔ copilot backend) should be able to spawn **sub-agents**: autonomous agents that work like the main assistant but with a different personality — either a **backend specialist** (backed by its own Kiln Task on the server, like `kiln-chat` is today) or a **throwaway** agent (thin stock task + fresh prompt seeded by the parent). The parent must be able to track sub-agent progress and merge their results back into its own reasoning.

**Decisions made with the user:**
- Sub-agent loops run **locally in the desktop process** (reusing the auto-mode machinery). To the backend, each sub-agent is an ordinary chat session — this is forced by the client-tool protocol: `call_kiln_api` must execute against the user's local server, and the wire protocol is client-driven.
- Agent personalities come from a **backend specialist registry** + a generic throwaway type.
- Progress: **poll (status tools) + completion injection** of the child's final report into the parent.
- Consent: **one-time user consent at spawn**, then sub-agents auto-approve their tools (like auto mode). Auto-mode parents have implied consent.
- Depth 1 in v1: sub-agents cannot spawn sub-agents (enforced on both sides).

Two repos change: `kiln_server` (backend, `/Users/leonardmarcq/dev/kiln_ws_3/kiln_server`) and `Kiln` (desktop, this repo).

## Architecture in one paragraph

A sub-agent is a locally-supervised autonomous chat loop (a sibling of `AutoChatRunner`) against the same upstream `POST /v1/chat/`, with its own trace chain, declaring its agent identity (`agent_type` + seed prompt + parent linkage) on its first turn. The parent model orchestrates via new client-executed tools (`spawn_subagent`, `get_subagent_status`, `wait_for_subagents`, `stop_subagent`) whose schemas the backend defines as unmanaged tools. The backend persists agent identity and parent/child lineage in chat snapshots (new `session_meta`), so history/lineage survive app restarts; the desktop registry owns runtime state, caps, and report delivery.

## Wire contract (locked between the two repos)

- **`ChatRequest` gains** `agent: AgentSessionRequest | None` where `AgentSessionRequest = {agent_type: str, seed_prompt: str | None, parent_trace_id: str | None}`. Honored **only on the first turn** — `agent` + `trace_id` together → 400. On resume, identity comes exclusively from the stored snapshot.
- **`ChatSnapshot` gains** `session_meta: ChatSessionMeta | None` = `{root_id, agent_type, depth, parent_root_id, spawned_from_trace_id, seed_prompt}`. Key insight (verified): `trace_id` rotates every turn (new snapshot id per persist), so lineage keys on a stable `root_id` minted at session creation. Legacy snapshots → `None` → default agent; mint-if-missing on resume. No migration.
- **Session endpoints expose** `agent_type`, `root_id`, `parent_root_id`, `is_subagent` on `ChatSessionListItem` and `GetChatSessionResponse`. Clients render the history tree by grouping on `parent_root_id`.
- **Orchestration tools** (client-executed; schemas defined backend-side as unmanaged tools `kiln_unmanaged::<name>`):
  - `spawn_subagent(agent_type: enum-from-registry, name: str, prompt: str)` → `{"status":"spawned","subagent_id":"sa_..."}` or `{"status":"declined"}` / error. `subagent_id` is a **desktop-minted handle**; the child's session `root_id`/trace id appear later via status results.
  - `get_subagent_status(subagent_id?: str)` → status array (omit id = all children of this session): `{subagent_id, name, agent_type, state: running|completed|failed|stopped|timeout, rounds, session_root_id?, report? (terminal, on request)}`.
  - `wait_for_subagents(subagent_ids: str[], timeout_seconds?: int)` → same array + `timed_out: [ids]`. **Default 120s, max 300s** (desktop clamps regardless — the interactive `POST /api/chat` stream has no keepalive; bounded block is safe on localhost only).
  - `stop_subagent(subagent_id)` → `{"status": "stopped"|"not_found"|"already_finished"}`.
  - `send_to_subagent` as a **model-facing tool is punted from v1** (stop-and-respawn is enough; adding it later is a server-only deploy). The **user** can still message a running sub-agent from the UI via a local desktop endpoint (see Desktop §5/§6) — that path needs no backend tool schema.
- **Spawn mapping (desktop)**: child's first POST = `{messages: [canned kickoff user message], agent: {agent_type, seed_prompt: prompt, parent_trace_id: <parent's current leaf trace_id>}, auto_mode: true}`. The backend appends `seed_prompt` to the agent task's instruction as an "# Operator briefing" section (in-memory only).
- **Consent transport**: backend marks `spawn_subagent` with `kiln_metadata.requires_approval = true` → it rides the existing `tool-calls-pending` approval flow → `POST /api/chat/execute-tools`. Desktop downgrades to auto-approve after the first approval per parent conversation (consent memory in the sub-agent registry, not the browser). Auto-mode parents: implied consent (add a bullet to the auto consent dialog).
- **Version gating**: new backend setting `chat_subagents_min_client_version`; older clients never receive the spawn tool definitions (so the model can't call them) and `ChatRequest.agent` is rejected 400. No 426 bump.
- **Depth**: backend rejects `depth > 1` at child creation (computed from parent snapshot meta); child agent types don't include spawn tools; desktop runner additionally short-circuits orchestration calls at depth ≥ 1.

## Top risks

1. **Two-repo contract drift** — tool names/schemas and `agent` block shape must land per this contract; desktop Phase B depends on Backend B being deployed (or a local backend).
2. **Cost amplification** — N children × server rounds with only client-side caps + a weak server backstop; a real per-user limiter is a flagged follow-up.
3. **Report-as-user-message ambiguity** — reports ride the user role; frame parsing on hydration must tolerate models quoting the frame. Migrate if the backend ever grows a dedicated channel.
4. **Interactive `wait` blocks silently** — no keepalive on `POST /api/chat`; clamped to 300s, UI shows the tool spinner; the strip's live status mitigates. Streaming-wait is v2.
5. **Vocabulary collision** — "sub-agents" already means "Kiln Task as Tool" in the skill; the disambiguation edits are load-bearing for model behavior.
